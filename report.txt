ABSTRACT

This paper describes the design and experimental evaluation of a multi-threaded
transaction processing layer built on top of the RocksDB embedded key-value
store. Two concurrency control protocols were implemented and compared:
Optimistic Concurrency Control (OCC) and Conservative Two-Phase Locking
(Conservative 2PL). Both protocols were evaluated across two workloads of
differing complexity under varying degrees of thread parallelism and key
contention. The results demonstrate that OCC tends to outperform Conservative
2PL under low contention, while Conservative 2PL handles moderate contention
more gracefully by avoiding the wasted work associated with transaction
restarts. At high contention, both protocols degrade, but through distinct
mechanisms that reflect their fundamental design trade-offs.


1. INTRODUCTION

Concurrency control is one of the most critical components of any database
system. When multiple transactions execute concurrently, they may read and
write overlapping data in ways that produce inconsistent or incorrect results.
The two most widely studied families of concurrency control are pessimistic
approaches, which serialize access to data before conflicts occur, and
optimistic approaches, which allow transactions to proceed freely and only
check for conflicts at commit time.

Both strategies have well-understood trade-offs. Pessimistic methods add
overhead on every access but avoid wasted work. Optimistic methods have low
overhead during execution but can cause entire transactions to be discarded and
restarted. The practical performance of each protocol depends heavily on how
often transactions actually conflict, which is a function of workload
characteristics: the number of concurrent threads, the size of the key space,
and how skewed the access pattern is.

This project builds a complete transaction processing layer on top of RocksDB
and implements both protocols. The system is evaluated using two workloads: a
simple account transfer workload and a TPC-C-inspired workload involving
warehouse, district, customer, and stock records. For each workload, experiments
vary the number of threads (1, 2, 4, 8) at a fixed contention level, and then
vary the contention level (0%, 30%, 70%, 100% hot-key probability) at a fixed
thread count. Throughput, average response time, tail latency, and retry rate
are reported for each configuration.


2. BACKGROUND

2.1 Optimistic Concurrency Control

OCC was introduced by Kung and Robinson [1] as an alternative to lock-based
concurrency control for workloads where conflicts are rare. The protocol
divides transaction execution into three distinct phases. In the read phase,
the transaction runs freely without acquiring any locks. All writes are
buffered locally and not made visible to other transactions. In the validation
phase, the system checks whether any data read by the transaction has been
modified by a concurrently committed transaction since it was originally read.
If validation succeeds, the transaction enters the write phase and its buffered
updates are applied to the database. If validation fails, the transaction is
aborted and must restart from the beginning.

The key strength of OCC is that transactions never block during execution.
Under low contention, most transactions will pass validation and commit without
any interference. The weakness is that when conflicts do occur, all of the
work done during the read phase is discarded and repeated, which can become
expensive under high contention.

2.2 Conservative Two-Phase Locking

Two-Phase Locking (2PL) is a classical pessimistic protocol in which a
transaction must acquire a lock on any data item before accessing it, and
cannot acquire new locks after releasing any lock [2]. The conservative variant
requires transactions to declare and acquire all locks they will need before
execution begins. If any required lock is unavailable, the transaction releases
all held locks and waits before retrying. This all-or-nothing acquisition
strategy eliminates deadlock entirely: since no transaction holds locks while
waiting, a circular dependency between waiting transactions cannot form.

The trade-off versus standard 2PL is that conservative 2PL requires knowing
the full access set of a transaction upfront. In practice this is achieved
through a dry-run phase. While livelock is theoretically possible (two
transactions perpetually failing to acquire the same set of locks simultaneously),
it is addressed in practice through randomized backoff.


3. SYSTEM DESIGN

3.1 Storage Layer

The storage layer is a thin wrapper around RocksDB, providing the standard
key-value operations: put, get, delete, and a full-scan iterator. RocksDB
was chosen for its mature performance characteristics and thread-safe I/O.
All concurrency control logic is implemented above the storage layer; RocksDB
itself is treated as a black box.

Database records are structured as maps of named fields, serialized to a
string format of the form {field: value, field: value, ...}. The Record class
handles serialization and deserialization, including quote-aware comma parsing
to correctly handle field values that may themselves contain commas.

3.2 Transaction Abstraction

A Transaction object represents a single in-flight transaction. It holds two
data structures: a write buffer and a read set. The write buffer is a map from
key to pending value. All writes performed by a transaction go into this buffer
and are not applied to the database until commit. The read set is a list of
(key, value) pairs recording what the transaction read from the database and
when. If a key is read that was already written in the same transaction, the
read is served from the write buffer without touching the database.

This design cleanly separates the transaction's view of the world from the
database state, which is required for correct OCC validation and for 2PL's
dry-run key discovery. On commit, flush_writes() iterates the write buffer and
applies each entry to RocksDB.

3.3 OCC Implementation

The OCC class manages a single validation mutex and atomic counters for
committed and aborted transactions. The protocol follows three phases:

  Read phase: A fresh Transaction object is created and the transaction
  function is executed. All reads record snapshots into the read set; all
  writes go to the local buffer. No locks are held at any point.

  Validation phase: The validation mutex is acquired, ensuring that only one
  transaction validates at a time. For each (key, snapshot) pair in the read
  set, the current value of that key is read from RocksDB. If the current value
  differs from the snapshot, or if the key has been deleted since it was read,
  a conflict is detected and validation fails.

  Write phase: If validation passes, flush_writes() commits the buffered
  updates to RocksDB, the committed counter is incremented, and the mutex is
  released. If validation fails, the transaction is aborted, the aborted
  counter is incremented, and after a brief fixed wait of 100 microseconds the
  transaction restarts from the read phase.

Using a single sequential validation mutex is sufficient to guarantee
serializability. The order in which transactions acquire the validation mutex
defines the serialization order, and the validation check ensures that each
transaction's read set is consistent with the state of the database at the
moment it commits.

3.4 Conservative 2PL Implementation

The TwoPL class maintains a hash map of per-key mutexes called the lock table.
Access to the lock table itself is protected by a separate map mutex to allow
safe concurrent insertion of entries for keys that have not been seen before.
All locks are exclusive, meaning both reads and writes require the same lock
type. This simplifies the lock manager considerably without affecting
correctness for these workloads.

The protocol proceeds as follows:

  Key discovery: The transaction function is executed once as a dry run using
  a temporary Transaction object. Because writes go to the buffer and are never
  flushed, this run has no side effects. The set of all keys accessed — the
  union of read and write keys — is collected after the dry run completes.

  Lock acquisition: The system attempts to acquire the mutex for every key in
  the access set using try_lock(), which returns immediately without blocking.
  If any lock is unavailable, all locks acquired so far are released and the
  transaction retries after a randomized backoff period drawn uniformly between
  100 and 500 microseconds. This random backoff is essential for livelock
  prevention: two transactions that repeatedly collide will desynchronize over
  time and eventually interleave successfully.

  Execution: Once all locks are held, the transaction function is executed for
  real. Reads and writes proceed normally through the Transaction abstraction.

  Release: flush_writes() commits the buffered updates to RocksDB, and then
  all locks are released. The committed counter is incremented.

The dry-run approach adds one extra execution of the transaction function per
lock attempt but keeps the lock manager stateless and simple.

3.5 Workload Runner and Contention Model

The WorkloadRunner class loads all database keys at startup and groups them by
prefix (e.g., A_* for accounts, W_* for warehouses). During execution it
spawns N threads, each running a fixed number of transactions selected uniformly
at random from the registered transaction templates.

Keys are selected using a hot-set contention model. The user specifies two
parameters: hot_prob, the probability that a given key selection is drawn from
the hot set, and hot_frac, the fraction of the key pool designated as hot.
In all experiments, hot_frac is fixed at 10%, meaning the hot set contains
the first 10% of keys in each pool. As hot_prob increases from 0 to 1,
transactions increasingly converge on the same small group of keys, driving
up conflict rates for both protocols.


4. EVALUATION

4.1 Experimental Setup

All experiments were run on [machine description]. Each workload configuration
executed 200 transactions per thread. Results reflect a single run per
configuration; no warm-up period was used, as the database is fully loaded
before timing begins.

Workload 1 involves a single transaction type: Transfer. Each Transfer reads
two account records and moves 1 unit of balance from one to the other (2 reads,
2 writes). The key pool contains 500 accounts (A_1 through A_500).

Workload 2 involves two transaction types drawn with equal probability:
NewOrder and Payment. NewOrder reads one district record and three stock
records, increments the district's order counter, and decrements stock
quantities (4 keys, 8 field updates). Payment reads one warehouse, one
district, and one customer record and updates financial fields on all three
(3 keys, 6 field updates). The key pools contain 8 warehouses, 80 districts,
2400 customers, and a stock table.

4.2 Retry Rate vs. Contention

[Figure 1: Retry rate (%) vs. hot_prob for OCC and 2PL — Workload 1]
[Figure 2: Retry rate (%) vs. hot_prob for OCC and 2PL — Workload 2]

At zero contention (hot_prob = 0.0), both protocols exhibit near-zero retry
rates. Transactions access uniformly random keys and the probability that any
two threads read or lock the same key simultaneously is low.

As hot_prob increases, retry rates rise for both protocols. OCC retries grow
because any two transactions that both read a key within the same validation
window will conflict if one commits first. The retry rate climbs sharply
because all transactions that overlap with the most recent committer must
restart, discarding their read-phase work.

Conservative 2PL retries rise for a different reason: two transactions
attempting to lock the same key at the same time will cause one to release and
back off. However, the backed-off transaction does not lose its execution
result — it simply waits. The retry overhead in 2PL is therefore lower cost
per retry than in OCC.

Workload 2 shows higher retry rates than Workload 1 at equivalent contention
levels. This is a direct consequence of larger transaction footprints: NewOrder
touches 4 keys and Payment touches 3, versus Transfer's 2. More keys per
transaction means a greater probability of overlap with any concurrent
transaction, increasing conflicts under both protocols.

4.3 Throughput vs. Number of Threads

[Figure 3: Throughput (txns/sec) vs. threads — Workload 1, hot_prob=0.5]
[Figure 4: Throughput (txns/sec) vs. threads — Workload 2, hot_prob=0.5]

With a single thread, both protocols perform similarly and establish a baseline
throughput. No concurrency means no conflicts, so neither validation nor
lock acquisition ever fails.

As thread count increases, throughput grows for both protocols initially but
the rate of improvement diverges. OCC's single validation mutex becomes a
serialization bottleneck: even though transactions execute in parallel during
their read phases, they must queue at the mutex to validate. The time spent
holding the mutex is short (a single pass over the read set), but under high
thread counts the queue builds up and reduces effective concurrency.

Conservative 2PL uses per-key locking rather than a global mutex, which allows
transactions that access different keys to commit in parallel. This gives 2PL
better scalability at higher thread counts when the key space is large enough
that transactions rarely collide. The cost is the additional dry-run pass over
the transaction function to discover the key set.

4.4 Throughput vs. Contention

[Figure 5: Throughput (txns/sec) vs. hot_prob — Workload 1, 4 threads]
[Figure 6: Throughput (txns/sec) vs. hot_prob — Workload 2, 4 threads]

OCC throughput degrades more steeply as contention rises. When hot_prob = 1.0,
all transactions draw keys from the same 50-key hot set (10% of 500), and
nearly every validation will find that its read set has been modified by a
concurrent committer. Each aborted transaction discards all work done during
its read phase and must repeat it, consuming CPU without producing commits.

Conservative 2PL degrades more gracefully because blocking transactions do not
lose work. A transaction that cannot acquire its locks simply waits and then
executes once. At hot_prob = 1.0 the system effectively serializes over the
hot set, reducing throughput toward the single-thread baseline, but without
the cascading restart behavior seen in OCC.

4.5 Response Time

[Figure 7: Avg response time (ms) vs. threads — Workload 1, hot_prob=0.5]
[Figure 8: Avg response time (ms) vs. threads — Workload 2, hot_prob=0.5]
[Figure 9: Avg response time (ms) vs. hot_prob — 4 threads, both workloads]
[Figure 10: P50/P95/Max response time distribution — Workload 2, 4 threads,
            hot_prob=0.5, NewOrder vs. Payment]

OCC response time is measured from the start of the first read phase through
the final successful commit, including all retry latencies. Under high
contention this can produce significant tail latency: while the average may
appear acceptable, transactions that are repeatedly aborted accumulate large
elapsed times before committing. The P95 and maximum response times for OCC
rise sharply as hot_prob increases.

Conservative 2PL response time is dominated by lock-wait duration. Each failed
lock attempt introduces a backoff of 100 to 500 microseconds, which is much
shorter than the time required to re-execute a full transaction. As a result,
2PL response times are more predictable: the distribution is tighter, with
P95 values much closer to the average than in OCC.

In Workload 2, NewOrder transactions showed slightly higher response times than
Payment transactions across all configurations. NewOrder accesses four keys
(one district and three stock records), giving it a larger lock footprint under
2PL and a larger read set to validate under OCC. Payment accesses three keys.
The difference in per-type response time is consistent with this difference in
transaction size.


5. CONCLUSION

This project implemented and evaluated OCC and Conservative 2PL on top of
RocksDB under a realistic multi-threaded workload model. The results confirm
the classical trade-off between the two approaches.

OCC is the better choice when contention is low. Its execution is entirely
lock-free, transactions almost never abort, and the sequential validation mutex
is only lightly contended. As contention rises, OCC degrades quickly because
aborted transactions discard all of their read-phase work. The tail latency
under high contention is a practical concern for applications with latency SLAs.

Conservative 2PL handles moderate contention better than OCC. Because blocked
transactions do not lose their execution results, the work-per-commit ratio
stays more favorable. The random-backoff retry strategy effectively prevents
livelock without requiring a separate deadlock-detection component. The main
cost of Conservative 2PL is the dry-run key-discovery pass, which executes the
transaction function an extra time per lock attempt, and the exclusive-only
locking policy, which blocks readers unnecessarily.

Future improvements could include read/write lock differentiation to allow
concurrent readers, a more adaptive backoff strategy for 2PL, and concurrent
OCC validation using multi-version timestamps rather than a global mutex.
